# 第5章-函数 面试题

关联：[[第5章-函数]]、[[00-Go语言学习索引]]

---

## Q1：Go 函数签名由什么决定？变量名影响签名吗？

**答：** 函数签名由参数类型列表和返回值类型列表决定，变量名不影响签名。两个函数只要参数和返回值类型一一对应，就是相同签名（相同类型）。

```go
func add(x int, y int) int { ... }
func sub(a, b int) int     { ... }
// 两者签名相同：func(int, int) int
```

---

## Q2：什么是 bare return？它的风险是什么？

**答：** 当函数所有返回值都有命名时，`return` 可以省略操作数，直接返回当前各返回值变量的值，称为 bare return。

```go
func f() (x, y int, err error) {
    x, y = 1, 2
    return // 等价于 return x, y, err
}
```

风险：多处 return 时不直观，读者难以判断每处 return 实际返回什么值，容易出错。不宜过度使用。

---

## Q3：Go 的参数传递是值传递还是引用传递？

**答：** 全部是值传递。传递 slice、map、指针、channel、函数时，传递的是这些引用类型本身的副本（即头部信息或指针的拷贝），但通过副本仍可修改被引用的底层数据。

注意：这不等于"引用传递"——函数内无法通过重新赋值参数变量来影响调用方的变量。

---

## Q4：Go 的递归会栈溢出吗？为什么？

**答：** 一般不会。Go 的 goroutine 使用可变栈，初始很小，按需自动增长，不受固定大小限制（大多数语言固定 64KB–2MB）。因此 Go 递归处理大量数据时不需要担心栈溢出问题。

## Q5：Go 错误处理有哪五种常见策略？

| 策略 | 场景 |
|---|---|
| 传播错误 | 子调用失败 → 直接返回给调用者 |
| 重试 | 偶发错误，加超时和退避 |
| 输出并退出 | 不可恢复，仅在 main 中用 `log.Fatalf` |
| 只记录日志 | 不影响主流程，继续运行 |
| 忽略 | 明确不影响逻辑，须加注释说明意图 |

构造错误链用 `fmt.Errorf("context: %v", err)`，错误信息不大写、不换行。

## Q6：函数值的零值是什么？调用零值函数会怎样？

**答：** 函数类型的零值是 `nil`。调用 nil 函数值会触发 panic。使用前应先判断：

```go
var f func(int) int
if f != nil {
    f(3)
}
```

函数值不可比较，不能作为 map key。

---

## Q7：什么是闭包？Go 中闭包捕获的是值还是引用？

**答：** 闭包是函数值 + 它引用的外部变量的组合。Go 闭包捕获的是变量的**引用**（内存地址），不是值的副本。因此外部变量的后续修改对闭包可见，闭包对变量的修改对外部也可见。

```go
func squares() func() int {
    var x int
    return func() int {
        x++
        return x * x
    }
}
f := squares()
f() // 1，f() // 4，f() // 9 — x 在调用间持续增长
```

变量的生命周期由引用它的闭包决定，不由词法作用域决定。

## Q8：循环变量闭包陷阱是什么？如何修复？

**答：** 在循环中将循环变量捕获进闭包，所有闭包共享同一个变量地址。循环结束后执行闭包，读到的是最后一次迭代的值。

```go
// 错误：所有闭包都删除同一个目录（最后一个）
for _, dir := range tempDirs() {
    rmdirs = append(rmdirs, func() { os.RemoveAll(dir) })
}

// 修复：在循环内创建局部副本
for _, dir := range tempDirs() {
    dir := dir // 新变量，每次迭代独立
    rmdirs = append(rmdirs, func() { os.RemoveAll(dir) })
}
```

同样的问题存在于 `go` 语句和 `defer` 语句中。

## Q9：可变参数 `...T` 和接收 `[]T` 的函数有什么区别？

**答：** 两者是不同类型的函数：

```go
func f(...int) {}  // 类型：func(...int)
func g([]int)  {}  // 类型：func([]int)
```

调用可变参数函数时可以直接传多个值，也可以用 `slice...` 展开；调用接收 slice 的函数必须传 slice。可变参数函数体内该参数表现为 slice，但类型签名不同。

## Q10：defer 的执行时机是什么？多个 defer 的执行顺序？

**答：** defer 语句在包含它的函数执行完毕时执行（return 正常返回或 panic 异常退出都会执行）。参数表达式在 defer 语句处立即求值，函数体延迟执行。多个 defer 按**逆序**（后进先出）执行。

```go
func f() {
    defer fmt.Println("first")
    defer fmt.Println("second")
    defer fmt.Println("third")
}
// 输出顺序：third → second → first
```

---

## Q11：defer 如何用于资源释放？标准惯用法是什么？

**答：** 获取资源后立即 defer 释放，确保所有代码路径（包括错误路径）都能释放资源：

```go
resp, err := http.Get(url)
if err != nil { return err }
defer resp.Body.Close() // 紧跟在获取资源之后

f, err := os.Open(filename)
if err != nil { return err }
defer f.Close()

mu.Lock()
defer mu.Unlock()
```

---

## Q12：defer 能修改函数返回值吗？

**答：** 能，但只有命名返回值才可以。defer 在 return 更新返回值变量后执行，deferred 匿名函数可以读写命名返回值变量：

```go
func triple(x int) (result int) {
    defer func() { result += x }()
    return double(x) // result = 2x，defer 执行后 result = 3x
}
fmt.Println(triple(4)) // 12
```

## Q13：循环中 defer 有什么坑？如何解决？

**答：** 循环体内的 defer 不会每次迭代执行，而是等函数返回才全部执行。处理大量文件时会耗尽文件描述符：

```go
// 危险：所有 f.Close() 在函数结束时才执行
for _, name := range filenames {
    f, _ := os.Open(name)
    defer f.Close()
}

// 正确：封装成独立函数
for _, name := range filenames {
    doFile(name) // 每次调用结束时 defer 立即执行
}
```

## Q14：panic 和 error 的适用场景有何区别？

| | `error` | `panic` |
|---|---|---|
| 适用场景 | 预期内的失败（I/O、网络、用户输入） | 不应发生的程序内部错误（bug、逻辑不一致） |
| 处理方式 | 调用方检查返回值 | goroutine defer 执行后程序崩溃 |
| 公有 API | 应返回 error | 不应依赖 panic |

`Must` 前缀约定：对于调用方明确知道输入合法的场景，包装函数在非法输入时 panic（如 `regexp.MustCompile`）。

## Q15：recover 在哪里调用才有效？有哪些使用原则？

**答：** `recover` 只有在 **deferred 函数内部**调用才能捕获 panic，其他位置调用返回 nil。

使用原则：
1. 只恢复自己明确定义的 panic 类型（如自定义 `bailout` 结构体），不要不加区分地恢复所有 panic。
2. 不要恢复其他包或回调函数引起的 panic。
3. 公有 API 应返回 error，不应用 panic/recover 作为正常控制流。

```go
func Parse(input string) (s *Syntax, err error) {
    defer func() {
        if p := recover(); p != nil {
            err = fmt.Errorf("internal error: %v", p)
        }
    }()
    // ...
}
```
